# RAG — Digital Arrest Scam Shield

Module owner: Person 4
Part of: Digital Public Safety Platform — ET AI Hackathon 2.0

## What this does

This is the retrieval layer behind the Scam Shield chatbot. The idea is
simple: most people who fall for digital arrest scams, OTP scams, or fake
UPI refund requests don't know what the actual playbook looks like, and by
the time they're suspicious it's often too late. This module lets the
chatbot ground its answers in real advisory content (CERT-In, RBI,
cybercrime.gov.in) instead of guessing — so when someone asks "is this
call legit," the response is backed by an actual source, not a generic
LLM answer.

Today's scope is just the pipeline: load documents, split them into
chunks, embed them, store them, and retrieve the most relevant ones for a
query. The chatbot/agent that sits on top of this is Day 2+.

## Why RAG instead of just prompting an LLM directly

A plain LLM will happily answer scam-related questions from its training
data, but that's exactly the kind of unverifiable output the brief flags
as risky for a citizen-facing tool — false positive rate has to be very
low, and answers need to be traceable back to a real source for the
"auditability" angle in the judging criteria. Retrieval-augmented
generation lets us cite which advisory a claim came from, instead of the
bot inventing something that sounds plausible.

## Stack

- LangChain (orchestration)
- ChromaDB (vector store, local persistence)
- sentence-transformers / MiniLM (embeddings — runs locally, no API key
  needed, so the whole team can run this without waiting on credentials)
- Python 3.10+

Embedding provider is swappable via env var (`RAG_EMBEDDING_PROVIDER`) in
case we later want to switch to Cohere or OpenAI embeddings for the demo
build — chosen this way so a provider change doesn't touch any other code.

## Setup

```bash
cd rag
python -m venv venv
venv\Scripts\activate        # source venv/bin/activate on Mac/Linux
pip install -r requirements.txt
cp .env.example .env
```

## Adding documents

Drop `.pdf`, `.txt`, or `.md` files into `documents/raw/`. See
`documents/SOURCES.md` for the list of advisory categories we're sourcing
from and which ones are still placeholders vs. real.

## Running ingestion

```bash
python -m src.ingestion.ingest
```

Loads everything in `documents/raw/`, splits it into overlapping chunks,
embeds each chunk, and persists the result to `vectordb/`. Re-run this any
time new documents are added.

## Querying

```python
from src.retriever.retriever import retrieve

results = retrieve("Is it safe to share an OTP over phone?", k=3)
for r in results:
    print(r.source, r.score, r.content[:120])
```

`retrieve()` is the function the rest of the team should call — Person 2's
backend will eventually wrap this behind an API route, and the Day 2
chatbot calls it directly. Signature shouldn't change without flagging it
to the team first, since other modules will depend on it.

## Tests

```bash
pytest tests -v
```

Covers document loading (including edge cases — empty files, duplicate
filenames, unsupported formats), chunking, embedding generation, and
retrieval. Embedding/vector-store tests skip rather than fail if run in an
offline environment, so CI doesn't block on network access.

## Folder guide

| Folder | What's in it |
|---|---|
| `documents/raw/` | Source advisory docs |
| `documents/SOURCES.md` | Source tracking — what's real vs. placeholder |
| `src/ingestion/` | `loader.py` (parsing), `ingest.py` (chunk → embed → persist) |
| `src/retriever/` | `retriever.py` — the search function |
| `src/utils/` | `embeddings.py` — embedding provider wrapper |
| `src/chains/`, `src/agents/`, `src/prompts/` | Reserved for Day 2 (chatbot, scam-pattern agent) |
| `configs/` | Settings, env vars, logging |
| `vectordb/` | Auto-generated by ingestion, gitignored |
